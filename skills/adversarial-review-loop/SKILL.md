---
name: adversarial-review-loop
description: Run adversarial review with cold-start reviewer subagents. Use when the user wants constructive critique, stress-testing, design review, repo review, plan review, spec review, CLI/workflow review, or help turning unclear decisions into clear questions and next steps.
---

The hardest problem in reviewing your own plan is that you can't. You read into it everything you meant, so it always looks sound to you. This skill solves that by hiring a skeptic with amnesia: a reviewer subagent that knows nothing except what you tell it, and whose job is to interrogate the plan until it either breaks or earns trust. Your job — the outer loop — is to run that interrogation well: answer what can be answered, escalate what can't, and keep the user oriented while two agents argue in the background.

Adversarial review needs subagents. If you can't spawn them in this runtime, stop and say so; a review you conduct against yourself is theater.

Begin by defining what's under review — a plan, a repo area, a document, a workflow, a decision — and gather what can be inspected directly. Read the code, run the commands, open the docs. Never ask the user for a fact you could discover yourself; every question forwarded upward that a grep could have answered erodes their confidence that the review is doing work.

Then create the reviewer. Give it minimal, task-local context: what it's reviewing, what it may inspect, what's out of scope, and that it's read-only unless told otherwise. Tell it to interrogate constructively, walk each branch of the design to a resolution, and above all: **one question at a time, each with a recommended answer, stopping after the single most important one.** The recommended answer matters more than it looks. It forces the reviewer to commit to a position instead of sniping from the sidelines, and it gives you something concrete to accept, refine, or refute. Start with two reviewers. Independent skeptics disagree in ways a single one can't, and where they diverge is usually where the plan is genuinely unsettled. Add more only when a distinct perspective would make a meaningful difference: one on the docs, one on the CLI behavior, one on repo structure. One is fine when the target is small enough that a second would just echo the first.

Now run the loop. When the reviewer asks its question, answer as the responsible agent. If the answer lives in the project, go look and answer from evidence. If it's a product or policy choice only the user can make, bring it to them — with the reviewer's recommendation and your own. Resolve dependencies in order: source-of-truth, ownership, naming, audience, and safety questions come before implementation details, because a downstream answer built on an unsettled upstream is wasted motion.

Watch which direction the plan moves as the review runs. Finding gaps is the point of the exercise; papering over every gap with new mechanism is its failure mode. A reviewer who spots ten holes and an outer loop that dutifully patches all ten will converge on something more elaborate and more fragile than what it started with. So treat every finding as having two honest resolutions, not one: build something to close the gap, or shrink the plan so the gap no longer exists. Cut the feature, narrow the scope, or declare the case out of bounds and say so where users will see it. Shrinking is not a shortcut, a hedge against risk, or a lack of ambition. It is design judgment: the target is a plan with exactly the complexity the goal requires, no more and no less, and removal is how you get there as often as addition is. Tell reviewers this too: a recommended answer of "delete this" or "don't handle it" carries the same weight as "add handling." And use growth as a signal. If each iteration makes the design bigger, the loop is compounding complexity instead of resolving it, and the next question to put to the reviewer is what can come out.

While this goes on, remember the user is watching a dense exchange between two agents speaking shorthand. After each significant branch, translate: define terms the first time they matter, prefer "what this means in practice" over abstract labels, and turn jargon into consequences. "The source repo is the list of what should exist; the installed folders are what the tools actually load; so we edit the source and let the sync command update the rest" — that sentence does more orienting than any status table.

Stop when the loop stops earning its keep: the key decisions are made and nothing higher-priority remains open; the next question hinges on a user preference no one can infer; the review has started repeating itself; or the user calls it. Before closing, ask the reviewer for a final concise critique and a recommended action list — the discipline of a closing statement often surfaces one last real objection. Then shut the reviewer down and report: what's now decided, what remains open, what should change next, and what evidence supports each conclusion. Present the intended end state cleanly, not the messy path of rejected ideas that led there. The user needs the map, not the wandering.
